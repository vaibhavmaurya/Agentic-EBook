# SOP 05 — Frontend Architecture Decisions

## Purpose

Frontend architecture choices lock in early and are expensive to change later. This SOP documents the key decisions — especially the static shell + runtime fetching pattern — and when to use each rendering strategy.

---

## The Most Important Decision: Rendering Strategy

Make this decision **before writing any frontend code**. Getting it wrong means a full rewrite.

### Option A: Static Site Generation (SSG) — Build-Time Data

All pages are pre-rendered at build time. Content is baked into the HTML files.

**When to use:**
- Content changes rarely (blog posts, documentation)
- You control when content updates (you trigger a build)
- SEO is critical and you cannot afford client-side rendering

**The critical constraint:** every content change requires a site rebuild and redeploy. If your backend generates content automatically (AI pipeline, user-submitted data), you need a rebuild webhook on every publish. This is fragile.

**When this fails:** the original public site used `[slug].astro` with `getStaticPaths()` to pre-render topic pages at build time. After every topic was published through the AI pipeline, an Amplify rebuild was triggered. This worked but had problems:
- Rebuild takes 2-3 minutes — content isn't live immediately after approval
- Amplify rebuild failure means content never goes live
- Worker Lambda needed `amplify:CreateDeployment` + `amplify:StartDeployment` IAM permissions
- More moving parts = more failure points

### Option B: Static Shell + Runtime API Fetching — Chosen Pattern

The site is a static shell (HTML + JS) deployed once. Content is fetched from the API at page load time in the browser.

**When to use:**
- Content changes frequently or unpredictably
- Content is generated by an automated system (AI pipeline)
- You want new content to appear immediately without a site rebuild
- Acceptable: content is client-rendered (JavaScript required)

**The trade-off:** pages are not pre-rendered — search engines see an empty shell until they execute JavaScript. For an admin-managed ebook platform, this is acceptable at MVP. Add SSR in Phase 2 if SEO becomes a requirement.

**Implementation in Astro:**

```astro
---
// topic.astro — reads ?id= from URL, fetches from API at runtime
// NO getStaticPaths() — this page is not pre-rendered
const API_URL = import.meta.env.PUBLIC_API_URL;
---

<script define:vars={{ API_URL }}>
  const params = new URLSearchParams(window.location.search);
  const topicId = params.get('id');

  async function loadTopic() {
    const res = await fetch(`${API_URL}/public/topics/${topicId}`);
    const data = await res.json();
    // render content...
  }

  loadTopic();
</script>
```

**Result:** new topics appear on the public site immediately after publish, with zero Amplify rebuilds.

### Decision Criteria Table

| Factor | SSG | Static Shell + Runtime Fetch |
|---|---|---|
| Content update speed | Minutes (rebuild) | Instant (API fetch) |
| SEO | Excellent | Poor (client-rendered) |
| Build complexity | Higher (data fetching at build) | Lower (shell never changes) |
| Infrastructure complexity | Build pipeline + CDN + redeploy webhook | CDN + API only |
| Good for AI-generated content | No | Yes |
| Good for blog/docs | Yes | Acceptable |

## Astro — Public Site Architecture

### Project Structure

```
apps/public-site/
  src/
    pages/
      index.astro        ← TOC home (fetches /public/toc at runtime)
      topic.astro        ← Topic viewer (?id= param, fetches at runtime)
      search.astro       ← Lunr.js search (fetches /public/search-index at runtime)
      releases.astro     ← Release notes (fetches /public/releases/latest at runtime)
    layouts/
      Base.astro         ← HTML shell, nav, head tags
    lib/
      api.ts             ← typed API client (all fetch calls go here)
      types.ts           ← TypeScript interfaces matching API response shapes
  astro.config.mjs
  package.json
```

### The `api.ts` Pattern

Never scatter `fetch()` calls across `.astro` files. Centralize all API calls in `lib/api.ts`:

```typescript
const API_URL = import.meta.env.PUBLIC_API_URL;

export async function getToc(): Promise<TocEntry[]> {
  const res = await fetch(`${API_URL}/public/toc`);
  if (!res.ok) throw new Error(`TOC fetch failed: ${res.status}`);
  return res.json();
}

export async function getTopic(topicId: string): Promise<TopicContent> {
  const res = await fetch(`${API_URL}/public/topics/${topicId}`);
  if (!res.ok) throw new Error(`Topic fetch failed: ${res.status}`);
  return res.json();
}
```

This makes API endpoint changes a one-file edit, not a search-and-replace across 10 pages.

### Lunr.js Search Integration

```typescript
// Fetch the pre-built index from the API, initialize Lunr, query it
const indexData = await getSearchIndex();
const idx = lunr.Index.load(indexData);
const results = idx.search(query);
```

The search index is rebuilt by `search_index_worker.py` on every topic publish. Fetching it at runtime means search is always up to date without a site rebuild.

### Amplify Environment Variables

Astro uses `PUBLIC_` prefix for variables exposed to the browser:

```bash
# In Amplify console / Terraform amplify_app env block
PUBLIC_API_URL=https://gcqq4kkov1.execute-api.us-east-1.amazonaws.com
```

Access in Astro: `import.meta.env.PUBLIC_API_URL`

**These are baked into the static JS bundle at build time** — they are NOT fetched at runtime. So changing the API URL requires a site rebuild. This is acceptable; API URLs don't change frequently.

### Cognito Auth Pattern

```typescript
// lib/auth.ts
import { signIn, fetchAuthSession } from 'aws-amplify/auth';

export async function getJwtToken(): Promise<string | null> {
  try {
    const session = await fetchAuthSession();
    return session.tokens?.idToken?.toString() ?? null;
  } catch {
    return null;
  }
}
```

### Axios JWT Interceptor

Never manually add `Authorization` headers to every API call. Use an Axios interceptor:

```typescript
// lib/api.ts
const api = axios.create({ baseURL: import.meta.env.VITE_API_URL });

api.interceptors.request.use(async (config) => {
  const token = await getJwtToken();
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

api.interceptors.response.use(
  (res) => res,
  async (error) => {
    if (error.response?.status === 401) {
      // Token expired — try refresh, then redirect to login
      await signOut();
      window.location.href = '/login';
    }
    return Promise.reject(error);
  }
);
```

### Zustand for Auth State

```typescript
// stores/authStore.ts
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

interface AuthState {
  user: User | null;
  setUser: (user: User | null) => void;
}

export const useAuthStore = create<AuthState>()(
  persist(
    (set) => ({ user: null, setUser: (user) => set({ user }) }),
    { name: 'auth-storage' }
  )
);
```

`persist` middleware stores auth state in `localStorage` so the user stays logged in across page refreshes.

### TanStack Query for Server State

Use TanStack Query for all API calls — not `useState` + `useEffect`:

```typescript
// TopicListPage.tsx
const { data: topics, isLoading, error } = useQuery({
  queryKey: ['topics'],
  queryFn: () => api.get('/admin/topics').then(r => r.data),
  staleTime: 30_000,  // re-fetch after 30s
});
```

Benefits: automatic caching, background refresh, loading/error states out of the box.

## Build Verification

**Always run `npm run build` before committing.** A build that works in dev may fail in build due to:
- TypeScript type errors (`strict: true` catches things dev doesn't)
- Missing environment variable references
- Import paths that only work with Vite's dev server module resolution

```bash
# Public site
cd apps/public-site && npm run build
# Must complete with 0 TypeScript errors, 0 build errors

# Admin site
cd apps/admin-site && npm run build
# Check bundle size — if > 1MB, investigate tree-shaking
```

## Amplify Deployment

### Manual Deploy (no GitHub integration)

```bash
# Build locally
npm run build

# Deploy to Amplify (uses zip artifact method)
aws amplify create-deployment --app-id <app-id> --branch-name main
# Returns: jobId, zipUploadUrl

# Upload build artifact
zip -r build.zip dist/
curl -T build.zip "<zipUploadUrl>"

# Start deployment
aws amplify start-deployment --app-id <app-id> --branch-name main --job-id <jobId>
```

The `publish_worker.py` handles this sequence after every topic approval, rebuilding and deploying the search index + TOC updates.

### Environment Variables in Terraform

```hcl
resource "aws_amplify_app" "public_site" {
  name = "ebook-public-${var.environment}"

  environment_variables = {
    PUBLIC_API_URL = "https://${var.api_gateway_id}.execute-api.${var.aws_region}.amazonaws.com"
  }
}
```

## Common Frontend Mistakes

| Mistake | Symptom | Fix |
|---|---|---|
| Using SSG for AI-generated content | Site rebuild required on every publish | Switch to runtime API fetching |
| Scattered `fetch()` calls | Hard to change API URL or add auth | Centralize in `lib/api.ts` |
| Manual JWT headers | 401 errors when token expires | Axios interceptor with auto-refresh |
| `useState` + `useEffect` for server data | Race conditions, duplicate fetches | TanStack Query |
| Committing without `npm run build` | TypeScript errors discovered in CI | Always build before commit |
| Hardcoding API URL | Works in dev, broken in prod | `import.meta.env.PUBLIC_API_URL` |
